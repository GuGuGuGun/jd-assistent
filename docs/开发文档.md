## 1. 项目概述

### 1.1 产品定位

基于 Web 端的 **智能简历优化系统**，用户上传原始简历 + 目标 JD（岗位描述），系统自动输出一份 **高度适配目标岗位** 的优化简历。

### 1.2 核心价值

| 维度 | 说明 |
|------|------|
| **精准匹配** | 通过 JD 关键词拆解，让简历与岗位需求高度对齐 |
| **事实可控** | 内置审查 Agent，交叉比对原始信息，拦截 AI 幻觉 |
| **一键成稿** | 从原文到终稿全流程自动化，输出可直投的 PDF |
| **数据复用** | 提取的结构化 `UserProfile` 可复用于面试辅导等下游模块 |

### 1.3 架构远景

系统前端节点（画像构建者）提取的 `UserProfile` 将作为底层数据持久化存储，未来可无缝对接 TechSpar 等 AI 面试辅导模块，实现 **"简历 → 面试"** 的数据闭环。

## 5. 全局状态定义

所有 Agent 节点的输入输出均收敛于 `ResumeGraphState`，由 LangGraph 统一管理。

### 5.1 State 文件 `backend/graph/state.py`

```python
from typing import TypedDict, List, Optional

class ResumeGraphState(TypedDict):
    """LangGraph 全局状态，所有节点的数据总线。"""

    # ═══ 用户输入 ═══
    original_resume_text: str          # 用户上传的原始简历纯文本
    target_jd_text: str                # 目标岗位的 JD 纯文本

    # ═══ Node 1 产出 ═══
    user_profile: Optional[dict]       # 结构化能力画像 (UserProfile)

    # ═══ Node 2 产出 ═══
    jd_analysis: Optional[dict]        # 岗位需求分析 (JobDescriptionAnalysis)

    # ═══ Node 3 产出 ═══
    optimized_contents: List[dict]     # 优化后的经历数组 (OptimizedExperience[])

    # ═══ Node 4 产出 ═══
    review_feedback: Optional[str]     # 审查员的修改意见（为空则通过）
    review_passed: bool                # 审查是否通过

    # ═══ 系统控制 ═══
    retry_count: int                   # 当前重试次数（上限 3）

    # ═══ 最终交付 ═══
    final_resume: Optional[dict]       # 终审后可渲染的完整简历 JSON
```

### 5.2 初始化默认值

```python
INITIAL_STATE: ResumeGraphState = {
    "original_resume_text": "",
    "target_jd_text": "",
    "user_profile": None,
    "jd_analysis": None,
    "optimized_contents": [],
    "review_feedback": None,
    "review_passed": False,
    "retry_count": 0,
    "final_resume": None,
}
```

## 6. 核心 Agent 节点详细设计

### 6.1 Node 1: 画像构建者 (Profile Builder)

| 属性 | 说明 |
|------|------|
| **文件** | `backend/agents/profile_builder.py` |
| **输入** | `state["original_resume_text"]` |
| **输出** | 更新 `state["user_profile"]` |
| **职责** | 从原始简历中提取客观能力画像，构建结构化的用户档案 |

#### 输出 Schema

```python
class UserProfile(BaseModel):
    """用户能力画像 — 仅提取客观事实，不做任何润色。"""

    name: str                                      # 姓名
    contact: dict                                  # 联系方式 {email, phone, ...}
    years_of_experience: float                     # 工作年限
    education: List[dict]                          # 教育背景 [{school, degree, major, year}]
    skill_matrix: dict[str, List[str]]             # 技能矩阵
    # 示例: {"frontend": ["Vue", "React"], "backend": ["Python", "Node.js"]}
    raw_experiences: List[dict]                    # 原始工作经历（逐条保留）
    # 每条: {company, title, duration, responsibilities: [str], achievements: [str]}
    certifications: Optional[List[str]] = None     # 证书 / 资质
    languages: Optional[List[str]] = None          # 语言能力
```

#### 关键实现逻辑

1. 将完整简历文本注入 Prompt，要求 LLM **仅提取事实，禁止推断**
2. 输出必须符合 `UserProfile` Schema（经过三重防线校验）
3. `raw_experiences` 中的每条经历保留原始描述，不做任何改写
4. 此节点的输出将持久化存储，用于后续面试辅导等下游模块

---

### 6.2 Node 2: JD 分析师 (JD Analyst)

| 属性 | 说明 |
|------|------|
| **文件** | `backend/agents/jd_analyst.py` |
| **输入** | `state["target_jd_text"]` |
| **输出** | 更新 `state["jd_analysis"]` |
| **职责** | 拆解 JD 中的硬性要求、软技能及业务痛点 |
| **并行** | 与 Node 1 并行执行，互不依赖 |

#### 输出 Schema

```python
class JobDescriptionAnalysis(BaseModel):
    """岗位需求深度拆解。"""

    job_title: str                                # 岗位名称
    company_info: Optional[str] = None            # 公司/团队信息
    must_have_skills: List[str]                   # 硬性技能要求
    nice_to_have_skills: List[str]                # 加分项技能
    pain_points: List[str]                        # 业务痛点 / 团队当前面临的挑战
    responsibility_keywords: List[str]            # 职责关键词（用于后续匹配）
    experience_range: Optional[str] = None        # 经验年限要求
    culture_keywords: Optional[List[str]] = None  # 文化/软技能关键词
```

#### 关键实现逻辑

1. 引导 LLM 从 "招聘方视角" 思考，提取显性和隐性需求
2. `pain_points` 需要从岗位职责描述中推断团队痛点
3. `responsibility_keywords` 用于 Node 3 关键词强制匹配

---

### 6.3 Node 3: 文案优化师 (Content Optimizer)

| 属性 | 说明 |
|------|------|
| **文件** | `backend/agents/content_optimizer.py` |
| **输入** | `state["user_profile"]`, `state["jd_analysis"]`, `state["review_feedback"]` |
| **输出** | 更新 `state["optimized_contents"]` |
| **职责** | 基于 STAR 法则重写经历，强制匹配 JD 关键词 |

#### 输出 Schema

```python
class OptimizedExperience(BaseModel):
    """单条优化后的工作经历。"""

    company: str                    # 公司名
    title: str                      # 职位名（可微调以匹配 JD）
    duration: str                   # 在职时间段
    highlights: List[str]           # 优化后的亮点描述（STAR 法则）
    matched_keywords: List[str]     # 本条经历命中的 JD 关键词
```

#### 关键实现逻辑

1. 遍历 `user_profile.raw_experiences`，逐条用 STAR 法则改写
2. 每条经历必须 **至少命中 1 个** `jd_analysis.must_have_skills` 关键词
3. 数字量化：优先保留原始数据中的量化指标；无量化数据时，**禁止凭空编造**
4. 若 `review_feedback` 不为空，Prompt 中追加反馈内容，要求针对性修正
5. 优化后的描述中对核心技术栈和关键数据加粗 (`**`)

---

### 6.4 Node 4: 内容审查员 (Content Reviewer)

| 属性 | 说明 |
|------|------|
| **文件** | `backend/agents/content_reviewer.py` |
| **输入** | `state["user_profile"].raw_experiences`, `state["optimized_contents"]` |
| **输出** | 更新 `state["review_passed"]`, `state["review_feedback"]` |
| **职责** | AI 幻觉拦截 — 交叉比对优化内容与原始信息 |

#### 审查规则

| 检查项 | 判定标准 | 严重程度 |
|--------|----------|----------|
| 技术栈捏造 | 出现原始简历中从未提及的技术 | 🔴 直接打回 |
| 指标夸大 | 量化数据偏离原始值 > 20% | 🔴 直接打回 |
| 职级虚高 | 职位名称实质性升级 | 🟡 警告并建议修正 |
| 时间线矛盾 | 在职时间与原始不符 | 🔴 直接打回 |
| 关键词堆砌 | 同一关键词在单条经历中出现 > 3 次 | 🟡 警告 |

#### 输出逻辑

```python
# 伪代码
if 所有检查项通过:
    state["review_passed"] = True
    state["review_feedback"] = None
else:
    state["review_passed"] = False
    state["review_feedback"] = "具体的问题描述和修改建议..."
```

---

### 6.5 Node 5: 终审排版员 (Final Typesetter)

| 属性 | 说明 |
|------|------|
| **文件** | `backend/agents/final_typesetter.py` |
| **输入** | `state["optimized_contents"]`（`review_passed` 必须为 `True`） |
| **输出** | 更新 `state["final_resume"]` |
| **职责** | 格式规范化 + 组装为前端可直接渲染的 JSON |

#### 输出 Schema

```python
class ResumeSection(BaseModel):
    """简历区块。"""
    type: str              # "summary" | "experience" | "education" | "skills" | "certifications"
    title: str             # 区块显示标题
    items: List[dict]      # 区块内容条目

class RenderReadyResume(BaseModel):
    """最终交付的简历 JSON，前端直接 map 渲染。"""
    name: str
    contact: dict
    summary: str                   # 个人总结（3-5 句话）
    sections: List[ResumeSection]  # 按渲染顺序排列的区块
```

#### 排版规范

- 核心数字加粗：`**提升了 40%**`
- 技术栈加粗：`**Vue 3**`, `**Python**`
- 统一英文大小写：`JavaScript` ✅，`javascript` ❌
- 日期格式统一：`2023.01 - 2024.06`
- 每条亮点描述控制在 1-2 行

## 7. LangGraph 工作流编排

### 7.1 路由规则

```
START
  ├──► Node 1 (画像构建者)  ─┐
  │                           ├──► Node 3 (文案优化师)
  └──► Node 2 (JD 分析师)  ──┘         │
                                        ▼
                                 Node 4 (内容审查员)
                                   │          │
                          通过 ───┘          └─── 打回（携带 review_feedback）
                            │                           │
                            ▼                           ▼
                     Node 5 (终审排版员)          返回 Node 3（重试，上限 3 次）
                            │
                           END
```

### 7.2 代码参考

```python
from langgraph.graph import StateGraph, END

def build_workflow() -> StateGraph:
    graph = StateGraph(ResumeGraphState)

    # 添加节点
    graph.add_node("profile_builder", profile_builder_node)
    graph.add_node("jd_analyst", jd_analyst_node)
    graph.add_node("content_optimizer", content_optimizer_node)
    graph.add_node("content_reviewer", content_reviewer_node)
    graph.add_node("final_typesetter", final_typesetter_node)

    # 起点：并行执行 Node 1 & Node 2
    graph.set_entry_point("profile_builder")
    graph.add_edge("profile_builder", "content_optimizer")

    # Node 2 也连接到 Node 3（LangGraph 会等待两个前置节点完成）
    graph.set_entry_point("jd_analyst")  # 第二个入口
    graph.add_edge("jd_analyst", "content_optimizer")

    # Node 3 → Node 4
    graph.add_edge("content_optimizer", "content_reviewer")

    # Node 4 → 条件路由
    graph.add_conditional_edges(
        "content_reviewer",
        route_after_review,
        {
            "pass": "final_typesetter",
            "fail": "content_optimizer",
        }
    )

    # Node 5 → 结束
    graph.add_edge("final_typesetter", END)

    return graph.compile()


def route_after_review(state: ResumeGraphState) -> str:
    """条件路由：审查通过则排版，未通过则打回重写。"""
    if state["review_passed"]:
        return "pass"
    if state["retry_count"] >= 3:
        # 超过最大重试次数，强制通过（记录告警日志）
        return "pass"
    return "fail"
```

## 8. Prompt 工程规范

### 8.1 Prompt 管理

- 所有 Prompt 存放在 `backend/prompts/` 目录下，**独立于代码**
- 使用 `.txt` 纯文本格式，支持 `{variable}` 占位符
- 通过 Python `str.format()` 或 `jinja2` 注入运行时变量

### 8.2 Prompt 编写原则

| 原则 | 说明 |
|------|------|
| **角色锚定** | 每个 Prompt 开头明确设定 Agent 角色，如 "你是一位资深的技术简历文案专家" |
| **约束前置** | 在 Prompt 开头列出所有硬性约束（如 "禁止捏造技术栈"） |
| **输出格式固化** | 给出明确的 JSON Schema 示例，减少格式偏差 |
| **Few-shot 示例** | 对复杂节点（如 Node 3），提供 1-2 个完整的输入输出示例 |
| **链式思维（CoT）** | Node 4 审查员使用 "请先列出每个检查项的判断依据，再给出结论" |

## 9. LLM 结构化输出防线

所有要求输出 Pydantic 模型的节点，**严禁直接调用普通 LLM 接口**，必须使用以下三重防线：

### 9.1 三道防线

```
┌──────────────────────────────────────────┐
│ 第一道：API 级防线                         │
│ response_format={"type": "json_object"}   │
└──────────────────┬───────────────────────┘
                   ▼
┌──────────────────────────────────────────┐
│ 第二道：物理级防线                         │
│ 正则提取：r'(\{.*\}|\[.*\])'              │
│ 剥离 ```json 等 Markdown 包裹符            │
└──────────────────┬───────────────────────┘
                   ▼
┌──────────────────────────────────────────┐
│ 第三道：代码级防线 (Agentic Retry)          │
│ 捕获 JSONDecodeError / ValidationError    │
│ 将错误信息拼回 Prompt，要求模型自我修正      │
│ 最大重试 3 次                              │
└──────────────────────────────────────────┘
```

### 9.2 参考实现

```python
import json
import re
from pydantic import BaseModel, ValidationError

async def safe_llm_generate(
    llm,
    prompt: str,
    schema: type[BaseModel],
    max_retries: int = 3,
) -> BaseModel:
    """三重防线 LLM 结构化输出生成器。"""

    for attempt in range(max_retries + 1):
        # 第一道防线：API 级 JSON 模式
        response = await llm.ainvoke(
            prompt,
            response_format={"type": "json_object"},
        )
        raw_text = response.content

        # 第二道防线：物理级 JSON 提取
        match = re.search(r'(\{.*\}|\[.*\])', raw_text, re.DOTALL)
        if not match:
            prompt += f"\n\n[系统错误] 上次输出未包含有效 JSON，请仅输出 JSON。"
            continue

        json_str = match.group(1)

        try:
            # 第三道防线：代码级校验
            data = json.loads(json_str)
            return schema.model_validate(data)

        except (json.JSONDecodeError, ValidationError) as e:
            if attempt < max_retries:
                prompt += f"\n\n[系统错误] 解析失败: {str(e)}。请修正后重新输出。"
            else:
                raise RuntimeError(
                    f"经过 {max_retries} 次重试仍无法生成有效的 {schema.__name__}"
                ) from e
```

### 9.3 R4 补充：Provider Fallback 与 Token 审计

R4 第一批后端实现已在 `backend/services/llm_service.py` 与 `backend/services/cost_calculator.py` 落地，目标不是改变业务 Prompt，而是补齐“可降级、可审计、可归因”的运行面。

1. **Provider / Model Fallback**
   - 主模型仍由 `LLM_PROVIDER` + `LLM_MODEL` 决定。
   - 当触发以下异常时，按 `LLM_FALLBACK_PROVIDERS` 顺序自动降级：API 超时、`429` 限流、`5xx` 服务端错误。
   - 默认降级链为：`GPT-4o → Claude-3.5-Sonnet → Gemini 1.5 Pro`。

2. **统一审计返回**
   - `safe_llm_generate(..., include_audit=True)` 会在原有结构化结果之外，补充返回 `provider`、`model`、`usage`、`attempts`、`fallback_used`。

3. **节点级成本归因**
   - 工作流节点执行完成后，会将审计信息写入 `Task.token_usage["nodes"][node_name]`。
   - 单次调用成本由 `backend/services/cost_calculator.py` 按模型单价估算，并落入 `cost_usd`。
   - 额度扣减审计继续保留在 `Task.token_usage["billing"]`，与节点级 token 审计并存。

### 10.4 当前任务状态持久化策略

- `tasks` 表负责保存任务主状态、节点日志、原始文件名、目标 JD、错误信息与完成耗时。
- `resumes` 表负责保存最终生成的 `RenderReadyResume` JSON，API 查询时再映射回 `result` 字段。
- `backend/services/event_bus.py` 负责抽象 SSE 事件传输边界：默认使用进程内总线，本地开发零配置可用；当启用 Redis 配置时，可切换到 Redis Pub/Sub 分发。
- `backend/services/checkpoint_store.py` 负责抽象 LangGraph checkpoint 存储：默认使用官方 `InMemorySaver`，在当前依赖集下保持本地零配置；当显式启用 Redis backend 时，会将单个 `thread_id` 的 checkpoint 快照持久化到 Redis，供后续 worker / 进程恢复使用。
- SSE 建连时会先根据数据库中的任务状态回放当前节点进度，再进入实时订阅，避免客户端断线重连后丢失当前视图。
- 任务执行派发由 `backend/services/task_dispatcher.py` 统一管理：默认 `local` 模式会在 API 进程内异步执行；切换到 `celery` 模式后，路由仅负责入队，实际执行交由 Worker 完成。
- Celery 任务会显式复用业务 `task_id` 作为 Celery `task_id`，因此数据库主键、状态查询接口与 SSE 回放链路都可以继续沿用同一标识。
- LangGraph 运行时也会显式复用业务 `task_id` 作为 `configurable.thread_id`，因此同一任务在不同进程间可以继续关联到同一份 checkpoint 数据。
- 任务创建时数据库主状态保持为 `pending`；只有本地协程或 Celery Worker 真正进入执行入口时，才会推进到 `processing`。对外 API 仍将 `pending` / `processing` 统一映射为 `processing`，因此前端与 SSE 契约无需变更。
- 当 Redis 不可用、未安装或连接失败时，SSE 传输层会自动安全回退到进程内总线；任务状态查询仍以数据库为准，因此现有 REST/SSE 契约保持不变。
- 当 LangGraph checkpoint Redis backend 未启用、未安装依赖或连接失败时，checkpoint 层同样会自动安全回退到内存模式；默认本地开发体验保持不变。

# ═══ LLM 配置 ═══

DATABASE_URL=sqlite+aiosqlite:///./jd_assistent.db  # 本地默认数据库；Docker Compose 可改为 ./data/jd_assistent.db

CELERY_BROKER_URL=redis://127.0.0.1:6379/1
CELERY_RESULT_BACKEND=redis://127.0.0.1:6379/2

# ═══ SSE 事件传输配置 ═══
SSE_EVENT_BUS_BACKEND=memory        # memory / redis，Docker Compose 场景建议切到 redis
REDIS_URL=redis://127.0.0.1:6379/0  # 未设置时会自动复用 CELERY_BROKER_URL
SSE_REDIS_CHANNEL_PREFIX=jd-assistent:sse

# ═══ LangGraph Checkpoint 配置 ═══
LANGGRAPH_CHECKPOINT_BACKEND=memory                 # memory / redis；默认 memory 保持本地零配置
LANGGRAPH_CHECKPOINT_REDIS_URL=redis://127.0.0.1:6379/0
LANGGRAPH_CHECKPOINT_REDIS_KEY_PREFIX=jd-assistent:checkpoint

# ═══ PDF 渲染 ═══
PUPPETEER_EXECUTABLE_PATH=           # Chrome/Chromium 可执行文件路径（可选）
```

说明：

- 默认推荐 `SSE_EVENT_BUS_BACKEND=memory`，适合本地开发与单进程部署。
- 默认推荐 `TASK_DISPATCHER_MODE=local`，适合 Windows、本地开发与默认测试环境；此模式不要求 Redis 或 Celery Worker。
- 默认推荐 `LANGGRAPH_CHECKPOINT_BACKEND=memory`，适合本地开发、测试与单进程运行；此模式不要求 Redis。
- R4 新增 `LLM_REQUEST_TIMEOUT_SECONDS` 与 `LLM_FALLBACK_PROVIDERS`，用于控制超时阈值与 provider/model 降级顺序。
- 若配置了 `ANTHROPIC_API_KEY` / `GOOGLE_API_KEY`（以及对应 `BASE_URL`），系统会优先使用 provider 级配置；否则回退到全局 `LLM_API_KEY` / `LLM_BASE_URL`。
- 如果需要把任务执行迁移到独立 Worker，可切换为 `TASK_DISPATCHER_MODE=celery`，并配置 `CELERY_BROKER_URL` / `CELERY_RESULT_BACKEND`。
- 如果需要多进程之间共享 SSE 实时事件，可再配合 `SSE_EVENT_BUS_BACKEND=redis`；若 Redis 暂时不可达，SSE 传输层会自动退回本地模式，不影响数据库中的任务状态。
- 如果需要在 API / Worker 之间共享 LangGraph checkpoint，可显式把 `LANGGRAPH_CHECKPOINT_BACKEND` 切到 `redis`；当前实现基于官方 `InMemorySaver` 加 Redis 线程快照，满足现有依赖集下的持久化与恢复钩子要求。
- 若未单独提供 `REDIS_URL`，系统会自动复用 `CELERY_BROKER_URL` 作为 Redis 事件总线地址，以减少容器化部署中的重复配置。
- 若未单独提供 `LANGGRAPH_CHECKPOINT_REDIS_URL`，系统会优先复用 `REDIS_URL`，再兜底复用 `CELERY_BROKER_URL`。
