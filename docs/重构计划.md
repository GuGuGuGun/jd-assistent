## 〇、重构文档审查意见

在正式制定计划前，先对 `重构文档.md` 提出以下审查反馈：

### 合理之处
1. **任务调度解耦 (§1.1)** — Celery + Redis 是成熟的 Python 异步任务方案，与现有 FastAPI 生态无缝集成。
2. **Redis Pub/Sub 替代内存 Queue (§1.2)** — 正确解决了 SSE 事件与 Worker 节点的耦合问题。
3. **PostgreSQL 数据持久化 (§2.1)** — 表结构设计覆盖了用户、任务、简历、画像四大核心实体，UserProfiles 的"能力画像资产池"概念为后续产品生态（如面试辅导）打下基础。
4. **JWT + OAuth 2.0 (§3.1)** — 标准做法，`Depends` 注入保护路由清晰可行。
5. **LLM 智能路由与降本 (§5)** — Token 审计与成本归因在 SaaS 场景中是刚需。

### 需要补充 / 调整之处
| 编号 | 问题 | 建议 |
|------|------|------|
| R1 | §1.2 提到 `AsyncPostgresSaver` 检查点，但 LangGraph 社区对此支持仍在演进中，稳定性待验证 | 初期用 Redis 做状态快照，后续切换到 PG Saver |
| R2 | §2.1 缺少数据库迁移策略 | 引入 **Alembic** 管理 DDL 版本 |
| R3 | §3.2 计费模块只描述了概念，未涉及实现路径 | 先落地 `credits` 扣减逻辑，支付集成放后期 |
| R4 | §3.3 "后台管理功能"只有一行标题，无内容 | 补充 Admin Dashboard 需求：用户管理、任务审计、系统监控 |
| R5 | §4 前端演进缺少状态管理方案 | 引入 **Pinia** 管理全局用户态/任务态 |
| R6 | 未涉及 **部署与 DevOps** | 补充 Docker 容器化 + CI/CD 流水线 |
| R7 | 未涉及 **测试策略** | 补充后端单元测试 + API 集成测试框架 |
| R8 | 未涉及 **数据安全与隐私** | 补充简历数据加密存储、GDPR 合规考虑 |

**ORM 模型设计**:

```python
# backend/db/models.py

class User(Base):
    __tablename__ = "users"
    id            = Column(UUID, primary_key=True, default=uuid4)
    email         = Column(String, unique=True, nullable=False)
    hashed_pw     = Column(String, nullable=True)       # OAuth 用户可为空
    auth_provider = Column(String, default="local")      # local / google / github / wechat
    credits       = Column(Integer, default=5)           # 初始赠送额度
    tier          = Column(String, default="free")       # free / pro
    created_at    = Column(DateTime, server_default=func.now())

class Task(Base):
    __tablename__ = "tasks"
    id            = Column(UUID, primary_key=True, default=uuid4)
    user_id       = Column(UUID, ForeignKey("users.id"), nullable=False)
    status        = Column(String, default="pending")    # pending / processing / completed / failed
    target_jd     = Column(Text)
    original_file = Column(String)                       # 原始上传文件名
    duration_ms   = Column(Integer, nullable=True)
    error_msg     = Column(Text, nullable=True)
    node_logs     = Column(JSONB, default=[])             # 各节点执行日志
    token_usage   = Column(JSONB, default={})             # prompt_tokens / completion_tokens 审计
    created_at    = Column(DateTime, server_default=func.now())
    completed_at  = Column(DateTime, nullable=True)

class Resume(Base):
    __tablename__ = "resumes"
    id            = Column(UUID, primary_key=True, default=uuid4)
    task_id       = Column(UUID, ForeignKey("tasks.id"), unique=True)
    user_id       = Column(UUID, ForeignKey("users.id"))
    render_data   = Column(JSONB)                        # RenderReadyResume 完整 JSON
    version       = Column(Integer, default=1)
    created_at    = Column(DateTime, server_default=func.now())

class UserProfile(Base):
    __tablename__ = "user_profiles"
    id            = Column(UUID, primary_key=True, default=uuid4)
    user_id       = Column(UUID, ForeignKey("users.id"), unique=True)
    skill_matrix  = Column(JSONB, default={})
    raw_experiences = Column(JSONB, default=[])
    education     = Column(JSONB, default=[])
    last_updated  = Column(DateTime, server_default=func.now())
```

**执行步骤**:
1. 安装依赖：`pip install sqlalchemy[asyncio] asyncpg alembic`
2. 创建 `backend/db/database.py`，配置 `create_async_engine` + `async_sessionmaker`
3. 创建 `backend/db/models.py`，定义上述 4 张表
4. 初始化 Alembic：`alembic init alembic`，配置 `env.py` 读取 `DATABASE_URL`
5. 生成首次迁移：`alembic revision --autogenerate -m "init tables"`
6. 在 `config.py` 新增 `DATABASE_URL = os.getenv("DATABASE_URL", "postgresql+asyncpg://...")`

```python
@celery.task(bind=True, max_retries=1)
def run_optimize_task(self, task_id: str, resume_text: str, jd_text: str):
    """
    原 resume_service.run_optimize_task 迁移至此，
    执行完毕后将结果写入 PostgreSQL，
    各节点事件通过 Redis Pub/Sub 推送。
```

```python
# backend/api/routes.py (改造后)
@router.post("/optimize")
async def submit_optimize(file: UploadFile, jd_text: str = Form(...), user = Depends(get_current_user)):
    resume_text = parse_resume_file(file)
    task = await db_create_task(user.id, jd_text, file.filename)
    run_optimize_task.delay(str(task.id), resume_text, jd_text)  # Celery 异步分发
    return {"task_id": str(task.id), "status": "pending"}
```

## 四、R3 — 商业化引擎 (Auth & Billing)

> **目标**: 支撑多租户隔离、用户认证、额度计费。

### R3.1 JWT 认证体系

**MVP 范围界定**:
- **本期必做**: 邮箱注册、邮箱密码登录、JWT 签发与校验、`/auth/me` 当前用户信息查询。
- **本期保护范围**: `/api/v1/optimize`、`/api/v1/export/docx/{task_id}`、任务历史查询等所有用户态业务接口。
- **延期项**: Google / GitHub OAuth 回调链路先保留接口设计，不作为 R3 首轮交付阻塞项。
- **会话策略**: 前端仅持有 access token，后端以 `Authorization: Bearer <token>` 做无状态鉴权。

**涉及文件变更**:
| 操作 | 文件路径 |
|------|----------|
| 新建 | `backend/auth/__init__.py` |
| 新建 | `backend/auth/jwt_handler.py` — JWT 签发/验证 |
| 新建 | `backend/auth/dependencies.py` — `get_current_user` Depends |
| 新建 | `backend/auth/oauth.py` — Google/GitHub OAuth 回调 |
| 新建 | `backend/api/auth_routes.py` — `/auth/register`, `/auth/login`, `/auth/oauth/callback` |
| 修改 | `backend/api/routes.py` — 所有业务路由添加 `Depends(get_current_user)` |
| 修改 | `backend/requirements.txt` — 新增 `python-jose[cryptography]`, `passlib[bcrypt]`, `httpx` |
| 修改 | `backend/config.py` — 新增 `JWT_SECRET`, `JWT_EXPIRE_MINUTES`, OAuth client IDs |

**核心接口**:
```
POST /api/v1/auth/register     # 邮箱注册
POST /api/v1/auth/login        # 邮箱+密码登录 → 返回 JWT
GET  /api/v1/auth/oauth/google # Google OAuth 重定向
POST /api/v1/auth/oauth/callback # OAuth 回调处理
GET  /api/v1/auth/me           # 获取当前用户信息
```

**关键实现约束**:
1. 注册时邮箱唯一校验失败应返回明确业务错误，而不是数据库原始异常。
2. 密码仅保存哈希值，不允许明文落库。
3. `get_current_user` 统一解析 JWT，未携带或 token 非法时返回 401。
4. OAuth 路由可先保留占位实现，但文档需明确标注“后续增量”。

**验收补充**:
- 新用户可完成注册并立即使用登录接口获取 JWT。
- 未登录访问受保护路由时，接口稳定返回 401。
- 已登录用户只能读取自己的任务与导出结果，不能跨用户访问。

### R3.2 额度计费模块

**MVP 范围界定**:
- **本期必做**: credits 查询、任务启动前额度校验、任务成功完成后扣减、额度不足阻断提交。
- **本期不做**: 在线支付、订阅扣款、发票、第三方支付回调。
- **计费目标**: 先把“能不能跑任务”与“任务完成后怎么扣额度”定义清楚，确保商业化基础闭环成立。

**涉及文件变更**:
| 操作 | 文件路径 |
|------|----------|
| 新建 | `backend/services/billing_service.py` — 额度扣减/查询逻辑 |
| 修改 | `backend/worker/tasks.py` — 任务开始前检查额度，完成后扣减 |
| 修改 | `backend/db/models.py` — User 表 `credits` 字段操作 |

**计费规则**:
- 每次完整 5 节点流转扣 1 credit
- Node 4 打回重试（≤3 次）不额外扣费
- Free Tier: 注册赠 5 credits，月度上限 3 次
- Pro Tier: 无限次数（按月订阅）

**扣费时机细化**:
1. 用户提交任务时，先校验当前 tier 与 `credits` / 月度次数是否允许继续。
2. 若任务在进入正式工作流前即因参数错误、文件解析失败而终止，则不扣费。
3. 若任务成功完成并产出最终结果，统一扣减 1 credit。
4. 若任务进入工作流后因平台异常失败，首版按“不扣费 + 记录失败原因”处理，避免误伤用户。
5. Node 4 审查打回导致的内部重试，由平台承担，不重复计费。

**建议补充数据字段**:
- `users.credits`：当前剩余额度。
- `users.tier`：当前订阅档位（`free` / `pro`）。
- 如需限制 Free 月调用次数，可在后续增加月度计数字段或独立账单流水表；首版可先以内聚服务逻辑实现。

**验收补充**:
- credits 充足时，任务可正常提交并在完成后正确扣减 1 次。
- credits 不足时，任务提交直接失败，并返回可读的额度不足提示。
- 任务失败、重试、重复轮询等场景不会造成重复扣费。

### R3.3 Admin 后台 API

**MVP 范围界定**:
- **本期必做**: 用户列表、任务审计列表、基础统计、人工调整额度。
- **本期不做**: 完整可视化后台页面、复杂 RBAC、支付对账、运营自动化。
- **角色约束**: 首版仅区分普通用户与管理员两类角色，由后端守卫统一校验。

**涉及文件变更**:
| 操作 | 文件路径 |
|------|----------|
| 新建 | `backend/api/admin_routes.py` — 管理端点 |
| 新建 | `backend/auth/admin_guard.py` — 管理员权限校验 |

**管理端点**:
```
GET  /api/v1/admin/users          # 用户列表 (分页、筛选)
GET  /api/v1/admin/tasks          # 任务审计 (按时间/状态/用户)
GET  /api/v1/admin/stats          # 系统统计 (DAU, 任务数, LLM 成本)
POST /api/v1/admin/users/{id}/credits  # 手动调整额度
```

**接口边界说明**:
1. `admin/users` 首版以分页、邮箱、tier、剩余额度为核心字段，不扩展过多业务视图。
2. `admin/tasks` 重点服务于审计与排障，应能按用户、状态、时间范围过滤。
3. `admin/stats` 首版只要求提供可直接支撑运营判断的聚合指标，无需做复杂 BI。
4. `admin/users/{id}/credits` 仅允许管理员人工加减额度，并要求记录调整来源与备注。

**验收补充**:
- 非管理员访问管理接口时返回 403。
- 管理员可以查询用户与任务列表，并能成功执行额度调整。
- 管理端点输出字段与前述 billing / auth 术语保持一致。

## 七、里程碑验收标准

| 里程碑 | 验收标准 | 验证方法 |
|--------|----------|----------|
| **R1** | PostgreSQL 表可读写；Docker Compose 一键启动；pytest 通过 | `docker compose up` + `pytest` |
| **R2** | 提交任务走 Celery Worker；SSE 事件从 Redis 中转；断点恢复可演示 | 提交任务后 kill worker，重启后继续执行 |
| **R3** | 用户可注册/登录；未认证请求被 401 拦截；额度扣减正确 | `curl -H "Authorization: Bearer ..."` 全链路 |
| **R4** | Dashboard 展示历史；主题可切换；LLM 降级日志可审计 | UI 操作 + 查看 Task.token_usage |

## 八、风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| LangGraph Checkpointer 兼容性 | R2 断点恢复可能失败 | 初期用 Redis 快照兜底，不强依赖官方 Saver |
| Celery Windows 开发环境兼容差 | 本地开发受阻 | 开发环境统一走 Docker / WSL2 |
| OAuth 第三方审核周期长 | R3 进度延迟 | 先实现邮箱注册+JWT，OAuth 路由保留但不阻塞首版上线 |
| 多模型 Prompt 行为差异 | R4 fallback 输出质量不一致 | 对每个 fallback 模型做 prompt 微调测试 |

## 九、建议的执行顺序

```
Week 1:  R1.1 (数据库) → R1.2 (Docker) → R1.3 (测试)
Week 2:  R2.1 (Celery) → R2.2 (Redis Pub/Sub)
Week 3:  R3.1 (JWT Auth) → R3.4 (前端登录) → R3.2 (计费)
Week 4:  R2.3 (断点恢复) → R3.3 (Admin)
Week 5:  R4.1 (Dashboard) → R4.2 (主题) → R4.3 (LLM路由) → R4.4 (审计)
```

**关键路径**: R1.1 → R2.1 → R3.1（数据库 → 任务队列 → 认证，串行依赖）
