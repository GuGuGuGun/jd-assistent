## 二、关键实现说明

### 2.1 Provider / Model Fallback

本轮先把 R4 的后端基础能力补齐，而不是直接做 Dashboard 空壳。`safe_llm_generate()` 现在支持主模型失败后的 provider fallback，当命中以下条件时会自动切换到下一个候选模型：

- API 超时
- `429` 限流
- `5xx` 服务异常

默认链路为：`GPT-4o → Claude-3.5-Sonnet → Gemini 1.5 Pro`。

### 2.2 Token 审计与成本归因

本轮新增了统一审计返回结构，节点侧可拿到：

- `provider`
- `model`
- `usage`（输入 / 输出 / 总 token）
- `attempts`
- `fallback_used`

执行链路会在节点完成后，将这份审计信息写入 `Task.token_usage["nodes"][node_name]`，同时通过 `cost_calculator.py` 补充 `cost_usd`。既有的 `token_usage["billing"]` 扣费审计保持不变，与节点级 token 审计并存。

### 2.3 为什么先做后端基础

R4 计划文档虽然把 Dashboard 放在前面，但当前仓库还没有完整的“用户历史任务 + token/cost + profile summary”数据面。如果先做前端，页面只能展示静态占位或重复返工。因此这轮先把后端审计契约补齐，让后续 Dashboard 可以直接接真实数据。

### 2.4 Dashboard 第一批切片

在后端数据面补齐之后，本轮继续完成了 Dashboard 的首个可用版本：

- 新增 `GET /api/v1/dashboard`，返回当前用户的摘要统计与最近任务历史。
- 新增前端 `/dashboard` 工作台页面，并在顶部导航中加入入口。
- Dashboard 首屏先聚焦真实可用信息：累计任务、已完成任务、处理中任务、累计 token / 估算成本。
- 历史任务列表直接消费后端真实字段，展示原始文件名、提交/完成时间、token 成本与失败原因，不额外引入图表依赖。

### 2.5 CreditChart / ProfileSummary 真实数据版

这轮继续把 Dashboard 从“摘要页”推进到“可读工作台”：

- 后端 `GET /api/v1/dashboard` 新增 `credit_chart` 与 `profile_summary`。
- `CreditChart` 使用真实任务审计数据，按日期聚合 `charged_credits / charged_tasks / llm_tokens / total_llm_cost_usd`，明确展示的是“真实消耗趋势”，不是伪造的余额历史。
- `ProfileSummary` 使用真实账户聚合与已持久化画像摘要。为此补上了 `profile_builder -> user_profiles` 的持久化链路，前端可读取 `top_skill_categories / experience_count / education_count / last_updated` 等真实字段。

### 2.6 ThemePicker 与多主题 token

为了让 Dashboard 与简历预览都能切换视觉风格，这轮新增了轻量主题系统：

- 新增 3 套主题 token：`deep-blue`、`forest-ink`、`sunset-paper`
- 新增 `theme` store，沿用现有 `Pinia + localStorage + hydrate` 模式持久化主题选择
- 新增 `ThemePicker`，放在应用壳层顶部
- Dashboard、TaskHistory、App Header/Footer、ResumePreview、TaskProcess 的关键表面色、强调色、边线与纸面色都改为令牌驱动
- 主题在应用挂载前就会生效，避免首次进入时出现明显闪烁

### 2.7 Credit Ledger 与真实余额历史图

这一轮把额度系统从“当前余额字段 + 任务扣费痕迹”升级成了真正的 credit ledger：

- 新增 `credit_ledger` 表，用于记录每一次额度增减、变动原因、变动后余额、关联任务与操作人。
- 新用户注册时会写入 `initial_grant` 流水，管理员调额会写入 `admin_adjustment` 流水，任务成功扣费会写入 `task_completion_charge` 流水。
- `User.credits` 仍保留为当前余额缓存，但所有真实余额变化都必须伴随 ledger 写入，因此后端现在同时具备“快速读取当前余额”和“可追溯的审计历史”。
- Dashboard 的 `CreditChart` 已从 `charged_usage` 切换为 `balance_history`，真实按 ledger 事件渲染余额历史，而不是再用任务消耗近似模拟。

### 2.8 并发一致性与注册原子性强化

根据第一轮 Oracle 复核，这轮继续补了两类关键强化：

- **注册原子性**：`/api/v1/auth/register` 现在会在同一个事务里同时创建 `users` 记录和 `initial_grant` ledger，避免出现“当前余额写入成功，但首笔流水缺失”的断档状态。
- **原子余额更新**：`billing_service.adjust_user_credits()` 与 `billing_service.finalize_task_charge()` 都改成数据库表达式更新（`credits = credits + delta` / `credits = credits - 1`），不再先读旧余额再写回，从而降低并发下丢失更新的风险。
- **并发回归测试**：新增并发管理员调额与并发任务扣费测试，确保累计结果正确、不会因为竞争条件出现超扣或丢账。

### 2.9 提交即预留额度

这一轮继续补上了产品语义层面的最后一块：从“完成时扣费”演进为“提交即预留、成功结算、失败释放”。

- 提交任务时，后端会在同一事务里完成：任务建档 + 额度预留 + `task_submission_reserve` ledger 写入。
- 任务成功时，不再额外扣减余额，而是把任务的 billing 状态从 `reserved` 结算为 `charged`。
- 任务失败时，若该任务之前已预留额度，则自动执行 `task_failure_release`，把额度退回用户余额，并把释放原因写入审计。
- `tasks` 表新增 `billing_status / billing_reserved_at / billing_released_at / billing_reservation_amount`，可以直接区分 `reserved / charged / released / waived` 等状态。
- 这样修完以后，“用户看到可提交就意味着额度已被占住”，不再出现多个任务抢同一份剩余额度、最后只有一部分能成功结算的语义断层。

### 2.10 预留竞态与派发失败兜底

在提交即预留落地后，Oracle 进一步指出了两个高优先级边界条件，这轮已经补上：

- **结算 / 释放竞态**：`finalize_task_charge()` 与 `release_task_reservation()` 现在都会先通过条件更新抢占 `reserved` 状态，只有抢占成功的一方才能继续改账。这样可以避免“先成功结算、随后又被失败释放反向加回额度”的错账问题。
- **派发即时失败**：如果任务预留成功后，`task_dispatcher.dispatch()` 立刻抛错，路由层现在会立即把任务标记为失败，并走统一的额度释放路径，确保不会遗留失败任务和被占用却未释放的额度。
- **新增回归**：补充了“结算与释放并发时账务仍一致”与“dispatch 立即失败时会释放额度”的自动化测试。
